# Engram Evaluation Harness — Gate E Baseline (2026-07-13)

## Overview

Two evaluation golden sets and a harness script under `evals/`:

- **Classification** (`evals/golden/classification_v1.json`): 20 samples covering
  all 8 builtin memory kinds (fact, preference, doctrine, decision, invariant,
  observation, procedure, summary). Each sample is a realistic memory write with
  an expected kind label.

- **Recall relevance** (`evals/golden/recall_v1.json`): 8 queries with relevant
  content fragments that should appear in top-K results. Measures precision@K,
  MRR, and recall@K.

## Baseline results

### Classification — rule-only (no LLM)

```
Accuracy: 20.0% (4/20)
```

Rule-only correctly classifies all 4 fact samples (via the "no rule matched"
conservative default) and nothing else. This is the floor — the rule engine has
no built-in keyword patterns for preference, doctrine, decision, etc. It relies
on tenant-configured regex rules + LLM enrichment.

### Classification — live (rule + LLM) on engram01 dogfood

```
Accuracy: 50.0% (10/20)
```

Per-kind: fact 100%, preference 100%, decision 67%, observation 50%,
doctrine 0%, invariant 0%, procedure 0%, summary 0%.

The live instance has some classification rules configured (preference detection)
that boost those categories. The LLM enrichment path was not exercised because
`classification_provider` is set to `none` on the dogfood — only rule-based
classification runs. Enabling LLM classification should significantly improve
doctrine, invariant, procedure, and summary detection.

### Recall — live on engram01 dogfood

```
Mean Precision@K: 0.025
Mean MRR:         0.062
```

Near-zero because the dogfood corpus contains only 2 items (the roadmap and a
probe). The recall eval requires a seeded corpus to produce meaningful numbers.
Seeding was blocked by a pre-existing DetachedInstanceError bug in the
classification vocab cache (see Known Issues below).

## Usage

```bash
# Rule-only classification (no instance needed)
python evals/run_evals.py --rules-only

# Full eval against a live instance
python evals/run_evals.py --base-url http://engram01:8000 --api-key eng_...

# JSON output for CI integration
python evals/run_evals.py --rules-only --json
```

## Known issues blocking better baselines

1. **DetachedInstanceError in classification vocab cache.** The
   `_load_vocab_cached` function stores ORM objects (MemoryKind instances) that
   outlive their session. When `/v1/remember` accesses them in a subsequent
   request, they're detached. Fix: convert to dataclass snapshots before
   caching (same pattern as `RuleSnapshot`).

2. **LLM classification not enabled on dogfood.** `classification_provider` is
   `none`. Enabling it (with an OpenRouter model) should dramatically improve
   classification accuracy for doctrine, invariant, procedure, and summary.

3. **Recall corpus too small.** The dogfood instance needs ~20-50 realistic
   memories for the recall golden set to produce meaningful precision/MRR
   numbers. Blocked by issue #1 (can't seed via the API).
